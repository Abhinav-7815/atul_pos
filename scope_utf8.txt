1. Executive Summary
Atul Ice Cream, established in 1990 and one of India's most beloved ice cream brands, requires a next-generation Point of Sale system that matches its premium brand identity. This document defines the complete scope for building an ultra-luxury, full-featured Django-based POS system designed to exceed the capabilities of existing solutions like Pet Pooja, Petoo, and Posist.
This POS system will serve as the operational backbone for all Atul Ice Cream outlets ΓÇö from standalone parlours to franchise chains ΓÇö delivering a silky, rose-gold-tinted UI with enterprise-grade functionality, real-time analytics, and AI-assisted insights. The system is purpose-built for the frozen desserts industry with deep understanding of its inventory cycles, seasonal demand patterns, and customer behavior.
2. Project Vision & Goals
Vision
To build the most beautiful and functionally superior POS system ever deployed in the Indian ice cream & quick-service restaurant (QSR) segment ΓÇö a system that staff love to use, managers trust completely, and owners can rely on for strategic decisions.
Primary Goals
Replace dependency on generic POS software like Pet Pooja with a brand-native solution
Deliver real-time, multi-outlet visibility for franchise owners and area managers
Reduce billing errors and revenue leakage through intelligent reconciliation
Build a loyalty ecosystem that rewards frequent customers and drives repeat footfall
Enable AI-driven demand forecasting to reduce ice cream wastage ΓÇö a critical metric
Provide a stunning, on-brand UI that reinforces Atul Ice Cream's premium identity
Success Metrics
Average billing time reduced from 3ΓÇô4 minutes to under 60 seconds
Inventory accuracy improved from ~80% to 99%+ with real-time deduction
Customer loyalty program enrollment rate > 40% of daily transactions
Wastage reduction of at least 20% via expiry tracking and AI forecasting
Zero billing discrepancies on daily reconciliation reports

5. Module Details
M-01: Cashier & Order Management
The heart of the POS system. Designed for single-touch speed ΓÇö a cashier should be able to bill a 3-item order in under 20 seconds.
Core Features
Full-screen touch-optimized order interface with visual product grid
Quick-add buttons for top-10 bestsellers per outlet
Multi-order session support ΓÇö hold and resume orders
Order types: Dine-In, Takeaway, Parcel, Delivery, Drive-Through
Split bill ΓÇö divide by items or custom amount across customers
Bill modification ΓÇö edit, delete line items before payment
Void and refund workflow with manager authorization OTP
Order notes per item (e.g., 'no cone', 'extra sprinkles')
Customer quick-attach ΓÇö look up loyalty member at billing time
Multi-currency support and multi-payment tender in one bill
Payment Methods
Cash with automatic change calculation
UPI (direct QR generation per order via Razorpay/PhonePe API)
Debit/Credit card via integrated EDC terminal
Loyalty points redemption (partial or full payment)
Prepaid Atul Ice Cream wallet top-up and redemption
Corporate invoice / credit accounts for bulk buyers
Coupon code and voucher redemption with real-time validation

Granular, role-based access control to ensure every user sees and can do exactly what their role requires ΓÇö nothing more.
Default Role Hierarchy

Owner/Director: financial reports, outlet comparison, strategy dashboards (read-heavy)
Area Manager: assigned outlets only, staff management, inventory oversight
Outlet Manager: own outlet full access, cannot modify pricing or system settings
Cashier/Billing Staff: order management, billing, customer lookup ΓÇö no admin access
Kitchen Staff: KDS view only ΓÇö no billing or inventory access
Inventory Manager: stock management, procurement, no billing access
Delivery Manager: delivery order queue, rider assignment
Permission Engine
Permission sets can be customized per user (role-override for exceptions)
Action-level permissions: view, create, edit, delete, approve, export
Module-level on/off toggles per role
Audit log: every create/update/delete recorded with user, timestamp, and IP
Two-factor authentication (TOTP) for Admin and Owner roles

7. Database Design Strategy
Database Engine
PostgreSQL 15+ as the primary database. Redis for caching, session management, and Celery task queue. Elasticsearch for product search and analytics aggregation.
Key Design Principles
Multi-tenancy via outlet_id foreign key on all transactional models ΓÇö no row-level security until needed
Soft deletes across all critical models (is_active / deleted_at) ΓÇö data is never hard-deleted
UUID primary keys for all public-facing models to prevent enumeration attacks
Audit mixin (created_by, updated_by, created_at, updated_at) on every model
Decimal(10,2) for all monetary values ΓÇö never float
JSON fields for flexible attributes (product extras, custom order metadata)
Database-level constraints on critical business rules (non-negative stock, valid date ranges)
Core Model Relationships
Outlet (1) ΓåÆ (M) Staff, Menu, Orders, Inventory
Order (1) ΓåÆ (M) OrderItem ΓåÆ (M) OrderItemModifier
Product (1) ΓåÆ (M) ProductVariant ΓåÆ (M) BOMIngredient ΓåÆ (1) Ingredient
Customer (1) ΓåÆ (M) Transaction, LoyaltyPoints, Feedback
StockBatch (M) ΓåÆ (1) Ingredient | tracked via FEFO
8. API Architecture
Django REST Framework (DRF) with drf-spectacular for OpenAPI 3.0 documentation. API versioning via URL namespace (/api/v1/, /api/v2/).
API Categories
Public API: Menu display, digital signage content (no auth)
POS API: Order management, billing, KDS ΓÇö JWT auth, rate-limited
Management API: Inventory, staff, analytics ΓÇö role-gated
Integration API: Swiggy/Zomato webhooks ΓÇö HMAC signature verification
Admin API: System configuration, tenant management ΓÇö IP-whitelisted
Real-Time Features
Django Channels + WebSocket for KDS live order updates
WebSocket for live dashboard auto-refresh (sales counter, queue display)
Server-Sent Events (SSE) for digital signage content updates
Celery + Redis for async tasks: report generation, email/SMS dispatch, AI inference
9. Security Framework
JWT authentication with 15-minute access tokens and 7-day refresh tokens
Role-based permission classes on every DRF viewset
TOTP two-factor authentication for admin accounts
Database-level encryption for PII fields (customer phone, email) using django-pgcrypto
HTTPS enforced everywhere; HSTS preloading
CSRF protection on all state-changing endpoints
SQL injection prevention via ORM parameterized queries only
Rate limiting per endpoint per user (django-ratelimit)
Complete audit trail ΓÇö every data mutation logged to audit_log table
PCI-DSS considerations: no raw card data stored; tokenization via payment gateway
GDPR/PDPA-ready: customer data export and deletion endpoints
10. Deployment Architecture
Cloud Infrastructure (AWS Recommended)
EC2 / ECS (Fargate): Django application servers in auto-scaling group
RDS PostgreSQL (Multi-AZ): managed database with automated daily backups
ElastiCache Redis: session, cache, Celery broker
S3 + CloudFront: static files and media with CDN
Application Load Balancer with SSL termination
Route 53: DNS management with health checks
On-Premise Outlet Setup
Each outlet runs a local POS client (Electron app or browser-based)
Offline-first: local SQLite cache for uninterrupted billing during internet outage
Auto-sync when connection is restored: conflict resolution via last-write-wins with flagging
Local receipt printing via direct ESC/POS commands ΓÇö no cloud dependency for printing
